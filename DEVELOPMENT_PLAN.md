# VidSnap — Development Plan

A local desktop tool that takes a video of any size and splits it into ~2-minute segments, saved as new video files **with zero quality loss**. Everything runs locally — no web server, no cloud, no accounts, **no telemetry**.

---

## 1. Core Technical Decision (read this first)

The single most important choice is **how the split is performed**:

| Approach | Quality | Speed | Cut precision |
|---|---|---|---|
| **FFmpeg stream copy** (`-c copy`) — *recommended* | Lossless (bit-identical video/audio streams) | Very fast (no re-encoding; a 2 GB file splits in seconds) | Cuts snap to the nearest keyframe, so segments are ~2:00 (e.g. 2:00–2:04) |
| FFmpeg re-encode | Slight generational loss (or huge files if near-lossless) | Slow (real-time or worse) | Frame-exact 2:00 cuts |

Because the requirement is "without losing video definition," **stream copy is the correct default**. Re-encoding is offered only as an optional "exact cut" mode with a clear warning.

The underlying command the tool wraps:

```
ffmpeg -i input.mp4 -c copy -map 0 \
       -f segment -segment_time 120 -reset_timestamps 1 \
       output_part_%03d.mp4
```

- `-c copy` — copy streams untouched (no quality loss)
- `-map 0` — keep all streams (video, all audio tracks, subtitles where the container allows)
- `-f segment -segment_time 120` — split every 120 seconds
- `-reset_timestamps 1` — each segment starts at 0:00 so it plays correctly everywhere

## 2. Tech Stack & Tooling

- **Language:** Python 3.12+, fully type-hinted
- **Project tooling:** `pyproject.toml` as the single config file; **uv** for environments, dependencies, and lockfile (`uv.lock` committed for reproducible builds); **Ruff** for linting *and* formatting; **ty** for type checking. All configured in `pyproject.toml` — no `requirements.txt`, no `setup.py`.
- **Engine:** FFmpeg + FFprobe — bundled static binaries, **pinned to an exact version and SHA-256-verified at download time** (see Phase 1). The user should not need to install anything.
- **GUI:** Tkinter (ships with Python, zero extra dependencies) with an explicit Windows DPI-awareness fix — file picker, segment-length field, progress bar
- **CLI:** Also exposed (`vidsnap input.mp4 --minutes 2`) for power users / scripting
- **Packaging:** PyInstaller **one-dir mode + Inno Setup installer** (one-file mode is routinely flagged as malware by AV heuristics). Fallback if Defender false positives persist: Nuitka. Code signing (~$200–500/yr cert) is deferred but documented as the known fix for SmartScreen warnings.
- **CI:** GitHub Actions from day one — lint + typecheck + tests on every push; installer built and smoke-tested on `windows-latest` on tagged releases.

### Licensing decision

Bundling FFmpeg makes its license our constraint. **Decision: use the GPL static build and license VidSnap under GPL-3.0.** Rationale: the GPL build includes libx264, which the optional "exact cut" re-encode mode needs; VidSnap is a free local tool, so GPL distribution terms cost us nothing. (The alternative — LGPL build + permissive project license — would mean dropping or redesigning exact-cut mode.) Record the FFmpeg build source (gyan.dev or BtbN), version, and checksum in the repo.

## 4. Development Phases

### Phase 1 — Environment, tooling & skeleton (½–1 day)
- Init git repo; `uv init` with the structure above; configure Ruff (lint + format) and ty in `pyproject.toml`; commit `uv.lock`.
- **CI pipeline now, not in Phase 7:** GitHub Actions running `ruff check`, `ruff format --check`, `ty check`, and `pytest` on every push. The packaging job is added in Phase 7, but the quality gates exist from the first commit.
- `scripts/fetch_ffmpeg.py`: download the **pinned** FFmpeg GPL static build, **verify its published SHA-256 checksum before unpacking** (fail hard on mismatch), place binaries in `bin/`. Runtime resolver: bundled binary first, fall back to `ffmpeg` on PATH. Version + checksum + source URL recorded in the script — updating FFmpeg (demuxer CVEs are regular) is a one-line PR.
- Add `CLAUDE.md`, `LICENSE` (GPL-3.0), `CHANGELOG.md`.
- Generate small test fixtures with `ffmpeg -f lavfi -i testsrc` (e.g. a 7-minute synthetic video → expect 4 segments); fixtures are generated by a test helper/CI step, not committed as binaries.

### Phase 2 — Probing module (½ day)
`probe.py`:
- Run `ffprobe -v quiet -print_format json -show_format -show_streams`.
- Extract: duration, container format, codecs, resolution, audio/subtitle stream count.
- Validate the input (exists, is a video, duration > segment length — if shorter, just copy/inform).
- Decide output extension: same container as input (`.mp4` → `.mp4`, `.mkv` → `.mkv`). Never transcode containers silently; MKV keeps everything, MP4 drops incompatible subtitle formats — warn if streams would be lost.

### Phase 3 — Splitting engine (1 day) — the core
`splitter.py`:
- Build the segment command from the Phase 1 template; parameterize segment length (default 120 s), output folder (default `<input name>_segments/` next to the input), and filename pattern (`<name>_part_001.<ext>`).
- **Subprocess safety:** FFmpeg/FFprobe are always invoked with an argument list — never `shell=True`, never string-concatenated commands — so hostile filenames cannot inject commands.
- Stream FFmpeg's `-progress pipe:1` output to compute % complete (out_time / total duration) and emit progress callbacks (consumed by both CLI progress bar and GUI).
- **Logging:** structured local logging (`logging` module) to a rotating file in the app data dir — command lines, timings, FFmpeg stderr on failure — so user-reported failures are diagnosable. Nothing ever leaves the machine.
- Handle errors: unreadable file, disk full, unsupported container for `-c copy` (rare) — surface FFmpeg's stderr in a readable message.
- Post-run verification: ffprobe each segment; sum of segment durations ≈ source duration (±2 s tolerance); non-zero file sizes.
- Optional **exact-cut mode** (Phase 6): re-encode with `-c:v libx264 -crf 17 -preset slow -c:a copy`, clearly labeled "slower, minor quality trade-off".

### Phase 4 — CLI (½ day)
```
vidsnap <input> [--minutes 2] [--out DIR] [--exact]
```
- Prints a summary (source duration → N segments), a progress bar, and the output folder path on completion.

### Phase 5 — GUI (1 day)
Tkinter window:
1. **DPI awareness first:** call `SetProcessDpiAwareness` (via ctypes) before creating the root window, or the UI renders blurry on modern high-DPI Windows displays.
2. "Choose video…" button (file dialog; also accept drag-and-drop via `windnd` if trivial, else skip)
3. Segment length spinner (minutes, default 2)
4. Output folder picker (defaults next to input)
5. Start button → progress bar + status text ("Segment 3 of 12…")
6. On finish: "Done — 12 files created" + "Open folder" button
- Run the split on a worker thread so the window stays responsive; disable Start while running; Cancel button kills the FFmpeg process and cleans up the partial segment.

### Phase 6 — Hardening & edge cases (1 day)
- Very large files (10 GB+): stream copy handles these fine; verify no memory growth (we never load the video into memory).
- Inputs shorter than 2 minutes → single copied output with a note.
- Filenames with spaces/unicode; paths on OneDrive (this project folder is OneDrive-synced — default output to the same drive, warn that sync of many large files may be slow).
- Variable frame rate videos, multiple audio tracks, rotated phone videos (rotation metadata is preserved by stream copy — verify with a fixture).
- Sparse-keyframe sources (e.g. screen recordings with 10 s+ GOPs): report actual segment lengths after splitting so drift is visible.

### Phase 7 — Tests & packaging (1 day)
- `pytest` suite: unit tests for command construction, integration tests splitting the synthetic fixtures and asserting segment count, durations, and codec equality (ffprobe codec of segment == codec of source ⇒ proof of no re-encode).
- **Packaging:** PyInstaller **one-dir** build including `bin/`, wrapped in an Inno Setup installer. One-file mode is explicitly avoided (AV heuristics flag the self-extracting pattern). If Defender still flags the build, switch to Nuitka before considering code signing.
- **CI release job:** on tag — build the installer on `windows-latest`, install it on the bare runner (this *is* the clean-machine smoke test), run `vidsnap --version` and a fixture split, upload the installer as the release artifact.
- README: install, usage, FAQ ("Why is a segment 2:03 and not 2:00?" → keyframe explanation), and an explicit privacy note: fully offline, no telemetry.

## 5. Timeline

Roughly **5–6 working days** of scope; with AI-assisted development expect **2–3 calendar days**. The phase *order* is the commitment, not the day counts: a usable CLI-only version exists after **Phase 4** and is worth using immediately.

## 6. Acceptance Criteria

1. Splitting a 1080p, 30-minute MP4 yields 15 segments, each playable in VLC/Windows Media Player, starting at 0:00.
2. `ffprobe` shows identical video codec, resolution, bitrate profile on segments vs. source (no re-encode occurred).
3. A 10 GB file splits without the app exceeding ~100 MB RAM.
4. Total runtime for stream-copy split of a 1-hour file: under ~1 minute on an ordinary laptop.
5. Cancelling mid-split leaves no corrupt files (partial segment removed).
6. CI is green (Ruff, ty, pytest) and the tagged release installs and runs a fixture split on a bare `windows-latest` runner.
7. The bundled FFmpeg version, source, and verified checksum are recorded in the repo.
